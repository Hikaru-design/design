# CLAUDE.md — Kakeibo (家計簿)

## Project Overview

A Japanese household budget (家計簿) single-page web app. Users track income and expenses by month, assign transactions to categories and payment cards, and visualize spending with charts. All data is persisted entirely in `localStorage` — there is no backend or database.

## Tech Stack

| Layer | Technology |
|---|---|
| Framework | Next.js 16 (App Router) |
| UI Library | React 19 |
| Language | TypeScript 5 (strict mode) |
| Styling | Tailwind CSS v4 |
| Component Library | shadcn/ui (style: `base-nova`, base color: `neutral`) |
| Charts | Recharts 3 |
| Icons | lucide-react |
| Linting | ESLint 9 with `eslint-config-next` |
| Deployment | Vercel |

## Core Data Models (`lib/types.ts`)

```ts
Transaction {
  id: string            // generated by generateId() — `${Date.now()}-${random}`
  date: string          // "YYYY-MM-DD"
  type: "income" | "expense"
  amount: number        // integer yen
  category: string      // Category.id
  cardId: string | null // Card.id, null for income
  memo: string
  createdAt: string     // ISO timestamp
}

Category { id, name, color, isDefault }
Card     { id, name, color, isDefault }
```

## Storage Layer (`lib/storage.ts`)

- Reads/writes to `localStorage` using three keys: `kakeibo_transactions`, `kakeibo_categories`, `kakeibo_cards`.
- All functions guard against SSR with `typeof window !== "undefined"`.
- Default categories and cards are seeded from `lib/constants.ts` when `localStorage` is empty.
- `isDefault: true` records cannot be deleted (filtered out in `deleteCategory` / `deleteCard`).

## Application Architecture

- **Single page** (`app/page.tsx`) owns all state: `transactions`, `categories`, `cards`, selected month/year, and dialog visibility flags.
- State is loaded from `localStorage` once on mount via `useEffect`.
- After any mutation, the relevant getter is called again to re-sync state from storage (e.g., `setTransactions(getTransactions())`).
- Month navigation via `prevMonth` / `nextMonth` helpers that handle year rollovers.
- The filtered `monthlyTx` array is derived inline by matching `t.date` against a `YYYY-MM` prefix string.

## Component Conventions

- Every component file begins with `"use client"` — this is a fully client-side app.
- Props are typed with local `interface XxxProps` at the top of each file.
- Helper functions (formatters, lookups) are plain `function` declarations inside the component function — not extracted unless reused.
- UI text is in Japanese throughout.
- Color is expressed as hex strings passed via `style={{ backgroundColor: color }}` (not Tailwind classes), because colors come from runtime data.

## Styling Conventions

- Tailwind CSS v4 — use utility classes directly; no `tailwind.config.js` (config is in `app/globals.css`).
- Design tokens use CSS custom properties from shadcn/ui (e.g., `bg-background`, `text-muted-foreground`, `text-destructive`).
- Layout: `max-w-2xl mx-auto px-4` — mobile-first, centered, max width 672 px.
- Component spacing uses `space-y-4` / `gap-3` patterns consistently.
- Dark mode is supported via the `.dark` class (shadcn's `@custom-variant dark`).

## Adding shadcn/ui Components

```bash
npx shadcn add <component>
```

Components are placed in `components/ui/`. The alias `@/components/ui` is configured in `components.json`.

## Path Aliases

`@/*` resolves to the repository root. Use this in all imports:
```ts
import { Transaction } from "@/lib/types";
import { Button } from "@/components/ui/button";
```

## Key Constraints

- **No server-side data** — do not add API routes or server actions unless explicitly requested. All persistence stays in `localStorage`.
- **No test suite** — there are currently no tests; do not assume a test runner is available.
- **No backend** — there is no database, authentication, or external API.
- **Japanese UI** — all user-facing strings should be in Japanese.
- **Default data is immutable** — never allow deletion of records with `isDefault: true`.
- **Single-page layout** — all views are rendered on `app/page.tsx` using conditional rendering and dialogs; do not add new routes unless explicitly requested.
